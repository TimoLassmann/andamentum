Epistemic System

A formal research system that takes a question and produces a thoroughly investigated, evidence-based answer with transparent confidence scoring and an HTML report.

The epistemic system works like a careful human researcher. It takes a research question, validates and clarifies it, searches academic databases and the web for evidence, groups findings into testable claims, checks each claim skeptically from multiple angles, and synthesizes the results into a structured HTML report. Every step is transparent: claims can only advance through stages when they pass deterministic evidence gates, and two final scores tell you how complete the investigation was and what the evidence says.

1. How It Works

The system runs through nine phases. There is no central orchestrator controlling the flow. Instead, declarative patterns match entity state and trigger operations. Work emerges from these patterns: when a claim is in a certain state, the matching operation fires automatically. This makes the pipeline self-organizing and traceable.

Key design decisions:
Clarify-validate loop: The question is clarified up to 3 times. Each clarification is validated by a separate agent that checks the subject, breadth, and constraints haven't drifted. If all rounds fail, the original question is used.
Pattern-driven scheduling: No orchestrator. Patterns match entity state and trigger operations. This makes the pipeline self-organizing.
Evidence-claim judgment: Every evidence-claim pair gets an explicit verdict (supports / contradicts / no bearing) before scoring.
Adversarial evidence storage: Counterarguments that pass a quality threshold are stored as Evidence entities with support_judgment="contradicts", deduplicated by source URL. This means adversarial findings participate in the posterior tally alongside regular evidence.
Evidence clustering: HDBSCAN groups similar evidence, selecting one representative per cluster. This avoids counting the same finding multiple times.
Unified dedup threshold: 0.7 cosine similarity across assertions, uncertainties, and caveats.
Deterministic gates: Promotion requires counted supporting sources, not subjective quality scores.

Clarification Validator

The pipeline begins with a clarify-validate loop. A clarification agent refines the research question, and a separate validation agent checks the result. The validator ensures three things: the clarified question covers the same subject, maintains the same breadth (no narrowing or broadening), and does not introduce constraints not present in the original. If validation fails, feedback is sent back to the clarifier for another attempt, up to 3 rounds. If all rounds fail, the original question is used as-is.

    V -->|"pass"| CQ["Use clarified question"]
    V -->|"fail + feedback"| C
    V -->|"3 rounds exhausted"| FO["Use original question"]

Truth Maintenance System (runs at any point)

The Truth Maintenance System (TMS, from Jon Doyle's work) is not part of the linear flow. It fires whenever evidence is invalidated, automatically cascading that change to all claims that depend on the affected evidence.

flowchart LR
    TRIGGER["Evidence invalidated\n(any phase)"] --> IE["Mark evidence +\ndependents invalid"] --> RC["Re-check affected\nclaims against gates"] --> OUTCOME{"Claim still\nqualifies for\nits stage?"}
    OUTCOME -->|"yes"| KEEP["Claim unchanged"]
    OUTCOME -->|"no"| DEMOTE2["Demote claim\n(triggers Peirce cycling)"]
When TMS fires: Evidence invalidation cascades to claims. If a claim's adversarial balance drops below 0.3, or contradicting evidence outweighs supporting evidence, the claim is revalidated and demoted if it no longer meets its stage gate. Demotion resets verification flags (this is "Peirce cycling") but persistent results like adversarial_balance are preserved — the investigation happened, the claim just needs to be reconsidered.

3. Verification Tracks

After a claim passes initial scrutiny, it is checked from multiple angles. Which checks run depends on the question type. Each track asks a different kind of question about the claim.

TrackWhat it doesPhilosophical basis
Adversarial searchActively look for evidence AGAINST the claim, like a devil's advocate. Generates counter-queries, searches for refutations, and computes an adversarial balance score. Counterarguments that pass a quality threshold are stored as Evidence entities with support_judgment="contradicts", deduplicated by source URL.Popper (falsification)
ConvergenceCheck whether evidence from independent methods, fields, or datasets agrees. If multiple independent lines of evidence point to the same conclusion, that's stronger than one line alone.Wimsatt (robustness)
Deductive validationCheck whether the claim's logical structure is sound. Are the premises valid? Does the conclusion follow?Hempel (deductive soundness)
ComputationalIf the claim can be tested with code (e.g., a statistical claim), run that test.Popper (falsification)
Argument analysisExamine the argument structure: identify premises, check for gaps, assess whether the argument is complete.Toulmin (argument quality)
Contrastive evaluationCompare competing explanations side by side. Which claim better explains the evidence? What observation would distinguish them?Lipton (inference to best explanation)
Consistency checkCheck whether claims contradict each other. If two claims from the same investigation conflict, flag the inconsistency.Kahneman (independent judgment)

How secondary tracks decide whether to fire: Each secondary track has a deterministic condition checked on the claim's state — no LLM call needed. 4. Claim Lifecycle

Claims start as hypotheses and advance through stages when they pass deterministic evidence gates. Each promotion requires meeting specific thresholds. If counter-evidence is found or scrutiny fails, claims are demoted back down. This creates a self-correcting cycle: claims can only reach high stages by surviving repeated scrutiny.

Supporting sources are counted as independent evidence items that have been explicitly judged as "supports" by the evidence-claim judge, are not invalidated, and are not corroborative duplicates. This is the core gate criterion: how many genuinely independent sources actively support this claim?

Demotion, Peirce cycling, and the investigation loop

When a claim fails scrutiny or is demoted (by TMS or by new counter-evidence), several things happen automatically:

  • Stage regression: The claim drops to a lower stage.
  • Verification flags reset: All track completion flags (adversarial_checked, convergence_checked, etc.) are cleared. The claim must be re-verified at its new stage. This is "Peirce cycling" — doubt triggers re-investigation.
  • Persistent results survive: Although flags are reset, values like adversarial_balance remain. The investigation happened; the results aren't erased.
  • Modification tracking: Each demotion increments the modification counter for Lakatos degeneracy detection.
  • Investigation loop: When scrutiny returns "needs_resolution", the claim enters an investigation cycle. The system generates targeted queries, searches for new evidence, and re-scrutinizes. Maximum 3 cycles before the claim is abandoned.
Degeneracy detection (Lakatos): If a claim has been modified more than 3 times total, or 3 or more times within 24 hours, it is flagged as potentially degenerative — meaning the theory is being patched rather than genuinely supported. Degeneracy blocks promotion.

5. Confidence Scoring

After the investigation completes, two deterministic scores are computed from the final state. No LLM calls, no trained weights, no tunable parameters. Both use the logistic function (a smooth S-curve that maps any number to a value between 0 and 1).

Answer Confidence — "How complete was the investigation?"

This is a process completion score. Each required epistemic check is binary: it either passed or it didn't. The score measures whether the system did its job thoroughly — not whether the answer is correct. Zero free parameters.

The formula counts passes and failures, then applies the logistic function:

confidence = 1 / (1 + e-(passes - failures))
Why logistic? The logistic function smoothly maps the integer difference (passes - failures) to a 0-1 scale. Equal passes and failures gives 0.5. Each additional pass or failure shifts the score predictably. No weights, no thresholds, no parameters to tune.

Posterior confidence — "What does the evidence say?"

Only computed for yes/no-style questions (verificatory, comparative, predictive). For other question types like "what factors affect memory?" there is no meaningful directional answer, so this score is not produced.

The posterior counts independent directional evidence across all active claims (including claims at HYPOTHESIS stage, so the metric is meaningful even when claims have not been promoted):

  • Each independent, non-invalidated, representative evidence item judged as "supports" counts as +1
  • Each independent, non-invalidated, representative evidence item judged as "contradicts" counts as -1
  • Evidence judged "no_bearing" or not yet judged is ignored
  • Adversarial counterarguments stored as contradicting Evidence entities count in the tally
posterior confidence = 1 / (1 + e-(claims supported - claims contradicted))

Terminology note: This metric was previously labelled “P(Y)” in early versions of the system. That notation was misleading for questions answered with “No” — e.g. “Will AI take all jobs?” answered “No” with high confidence would show P(Y) = 0.9997, which reads as “99.97% probability of Yes.” The metric is now labelled “posterior confidence” to make clear it measures confidence in the established claims, not the probability of a “Yes” answer.

Example — "Does homeopathy work?"

This is a verificatory question. After evidence collection, the system found 9 supporting evidence items (the studies cited by proponents). Adversarial search then found 28 contradicting sources (systematic reviews, meta-analyses showing no effect beyond placebo). These adversarial findings were stored as Evidence entities with support_judgment="contradicts".

Posterior confidence: log-odds = 9 - 28 = -19. Confidence = 1 / (1 + e19) ≈ 0.00%.

This extreme value correctly reflects that the overwhelming weight of independent evidence contradicts the claim. The answer confidence separately tells you whether the investigation process was thorough enough to trust this posterior.

7. Epistemic Traditions

The system implements ideas from seven named philosophical traditions. Each tradition maps to a concrete mechanism in the code. This is not decoration — each tradition solves a specific problem that a naive "search and summarize" system would miss.

TraditionPhilosopherWhat it meansSystem mechanism
Inquiry cycling C.S. Peirce Genuine doubt triggers investigation. The cycle is: doubt → investigate → gather new evidence → re-evaluate → either resolve the doubt or repeat. Scrutiny loop with up to 3 investigation cycles per claim. Demotion resets verification flags, forcing re-evaluation.
Falsification Karl Popper You cannot prove a claim true; you can only try to prove it wrong. Claims that survive serious attempts at refutation are stronger. Adversarial search: generates counter-queries, searches for disconfirming evidence, stores quality counterarguments as contradicting Evidence entities, computes a balance score.
Robustness William Wimsatt Results that hold across independent methods, datasets, or fields are more trustworthy than results from a single approach. Cross-domain convergence: classifies evidence by domain, checks pairwise independence, detects whether independent lines agree.
Deductive soundness Carl Hempel Even with strong evidence, a claim is weak if its logical structure is flawed. Check whether the premises support the conclusion. Deductive validation: examines argument structure, identifies logical gaps.
Belief maintenance Jon Doyle When one piece of evidence is invalidated, all beliefs that depended on it must be automatically reconsidered. Truth Maintenance System (TMS): invalidation cascades through the dependency graph. Claims are revalidated against stage gates.
Degeneracy detection Imre Lakatos A theory that keeps getting patched ("ad hoc modifications") without generating genuinely new predictions is degenerative. Modification counter: more than 3 changes total or 3 changes in 24 hours flags the claim as degenerative, blocking promotion.
Independent judgment Daniel Kahneman Each assessment should be made independently and then combined. Anchoring on a first answer biases later judgments. Each evidence-claim pair is judged independently by narrow agents. Evidence is clustered (HDBSCAN) before counting. Scores combine deterministically, not via LLM.

8. Agent Decomposition

Complex judgments are broken into sequences of narrow steps. Each narrow agent outputs 1-4 fields and makes a single focused judgment. Deterministic combination steps (clustering, counting, scoring) connect them. This prevents the kind of compounding errors that happen when you ask one LLM call to do everything at once.

Pattern-driven scheduling

There is no central orchestrator that says "first do X, then do Y." Instead, the system defines a set of patterns. Each pattern says: "when an entity is in state X, run operation Y." The scheduler queries entities, matches them against patterns, and executes the resulting work items.

This means the pipeline is self-organizing. If a claim gets demoted, its state changes, which re-matches earlier patterns, which triggers re-investigation. No special "demotion handler" is needed — the patterns already cover it.

Runner PromptedOutput fallback

When structured output validation fails after all retries (in tool-call mode), the runner retries once with PromptedOutput mode. This injects the JSON schema directly into the system prompt instead of relying on the model's tool-calling interface. The fallback exists because small models sometimes ignore tool definitions entirely, but will follow schema instructions embedded in the prompt. Only fires after the normal retry budget is exhausted.

Evidence provider routing

Provider selection uses semantic similarity between the research question and enriched provider descriptions. Each of the seven registered providers (OpenAlex, PubMed, bioRxiv, ClinicalTrials, ChEMBL, Monarch, OpenTargets) has a 2–3 sentence description with domain signals and example queries. The router embeds the clarified question and all provider descriptions using the same Ollama embedding backend used elsewhere in the pipeline, ranks providers by cosine similarity, and selects the top-K whose score exceeds a calibrated threshold (min_score=0.15). Web search is always appended as a universal fallback. The formulate_query agent then writes one optimized search query per selected provider.

Provider description embeddings are cached at module level (keyed by embedding model + provider name), so only the query embedding is computed per routing call.

This replaced a previous keyword-matching approach (DOMAIN_PROVIDER_MAP) which was brittle and could not handle queries outside its hardcoded vocabulary. The semantic router achieves 97.5% top-3 recall on a 200-query benchmark across all seven providers.

Claim creation modes

The system has two mutually exclusive modes for creating claims from evidence, selected by the claim_to_verify field on the Objective entity:

Research mode (default, claim_to_verify=None): The system explores evidence broadly and proposes novel claims via a 3-step decomposition pipeline — extract assertions from each evidence item, cluster assertions by semantic similarity (HDBSCAN), and draft one claim per cluster. This is the normal path for open-ended research questions like “What does the literature say about metformin and cardiovascular mortality?”

Verification mode (claim_to_verify set): The system creates a single Claim entity whose statement is the verbatim user-provided claim, linked to all gathered evidence. No LLM call, no clustering, no assertion extraction. The existing scrutiny, verification tracks, and promotion machinery then run on this exact claim. This mode is used for claim verification benchmarks (e.g., SciFact) where the task is to verify a specific pre-stated scientific claim, not to discover new claims.

In verification mode, if question_type is not already set, it defaults to “verificatory” so that the correct routing profile and posterior computation are applied.
