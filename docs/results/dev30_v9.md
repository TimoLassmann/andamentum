# dev30 v9 — pre-release benchmark snapshot

Frozen benchmark results pinned to the pre-release tag of the
`andamentum.epistemic` pipeline. These numbers are the operational
artifact of the architecture as it shipped — anyone reproducing the
release on the same corpus, same model, same providers should get
results within sampling noise of this table.

## Setup

- **Corpus**: SciFact dev30 (n=30 cases; n=20 calibratable after
  excluding NEI rows that lack a directional gold label).
- **Reasoning model**: `openai:gpt-5.4-nano`.
- **Embedding model** (for chunk + similarity): `embeddinggemma:latest`
  (local Ollama).
- **Providers active**: full registry (openalex, pubmed, europepmc,
  biorxiv, clinicaltrials, chembl, cochrane, monarch, open_targets,
  arxiv). Provider filtering is off; the dispatch agent self-selects
  per claim.
- **Routing**: description-driven dispatch (single per-provider
  agent). Investigation rounds use the same dispatch path with
  claim + angle as the input pair.

## Headline numbers (calibratable subset, n=20, with 95% bootstrap CIs)

| arm                          | AUC                | Brier              | ECE                | judge F1 |
|------------------------------|--------------------|--------------------|--------------------|----------|
| baseline (nano alone)        | 0.55 [0.23, 0.78]  | 0.354 [0.23, 0.48] | 0.405 [0.25, 0.60] | 0.362    |
| baseline_frontier (gpt-5.4)  | 0.84 [0.60, 1.00]  | 0.166 [0.05, 0.32] | 0.165 [0.07, 0.35] | 0.520    |
| rag (own retrieval)          | 0.81 [0.53, 0.97]  | 0.253 [0.12, 0.40] | 0.250 [0.11, 0.45] | 0.615    |
| rag_replay (shared evidence) | 0.69 [0.42, 0.94]  | 0.294 [0.16, 0.44] | 0.367 [0.21, 0.55] | 0.364    |
| **epistemic (nano)**         | **0.89 [0.66, 1.00]** | **0.166 [0.09, 0.25]** | **0.167 [0.07, 0.33]** | **0.449** |

## What this run supports

1. **Competitive with a frontier model on calibration metrics.**
   Epistemic AUC 0.89 vs frontier 0.84; Brier and ECE essentially
   identical. A small model (gpt-5.4-nano) using the epistemic
   pipeline matches the frontier model on its own.

2. **Architecture-attributable gain over RAG on the same evidence
   pool.** Compared to `rag_replay` (which consumes the same Evidence
   the epistemic pipeline gathered, but processes it via a standard
   summarise-then-judge RAG aggregation), the epistemic pipeline cuts
   Brier from 0.294 to 0.166 — a 40% reduction in squared posterior
   error. Same model, same evidence; only the reasoning architecture
   varies.

3. **Honest calibration.** ECE 0.167 means when the pipeline says
   0.85, it really is right about 85% of the time. The baseline (no
   retrieval, no structure) sits at 0.405 ECE — wildly overconfident.

## What this run does not support, and why

These items have been flagged honestly so any external reader can
form their own assessment.

1. **n=20 is small.** 95% CIs on the AUC numbers all overlap. The
   point-estimate advantage of epistemic over baseline_frontier
   (0.89 vs 0.84) is not statistically significant at this sample
   size. The claim "small + structured beats frontier" is suggestive
   here but requires larger n to defend. Recommended follow-up: n ≥
   100 on the calibratable subset.

2. **Run-to-run variance is substantial.** Across development versions
   v3–v9 the epistemic AUC ranged 0.74–0.93 on the same corpus.
   Single-run point estimates at n=30 are exposed to that variance.
   Repeated runs (3–5x) with mean and SE would tighten the claim.

3. **SciFact is plausibly in training data.** Frontier-model
   performance on SciFact reflects training-time exposure plus
   reasoning. The "epistemic + nano ≈ frontier" comparison only
   carries to genuinely-novel claims if it holds out-of-distribution.
   A post-training-cutoff or niche benchmark is needed to defend the
   architectural claim independently of memorisation.

4. **F1 is partially circular.** The synthesis text is generated by
   gpt-5.4-nano; the judge running F1 against the gold answers is a
   sibling LLM (gpt-4o-mini). Same-family models share priors about
   what "supportive language" looks like, so any system that lands in
   an unusual rhetorical register (e.g., the epistemic system's
   hedging — "the evidence supports … but does not establish …") gets
   penalised. F1 here is not a fair architecture-comparison metric.

5. **rag_replay is a particular RAG configuration.** It's the same
   model + the same evidence + a 4-5 sentence judgment prompt — a
   standard implementation pattern, not a strong baseline. The
   comparison is still controlled (same model, same evidence), but
   readers should know that rag_replay is not a sophisticated RAG
   architecture.

## Reproducibility

The release tag is the frozen reference point. Running the same
benchmark harness against the tagged commit, with the same model and
provider configuration, should produce results within sampling noise
of the table above. Cycle counts and provider distributions may vary
because dispatch agent decisions are stochastic at the prompt level.

## Cycling diagnostics

For the 12 cases that triggered ≥1 investigation round, the gap-
analysis agent's intents (visible per-claim in
`Claim.investigation_intents`) consistently shifted methodological
dimensions across rounds rather than paraphrasing the same lexicon.
Sample HCQ-trial-mortality case (3 rounds, 9 intents):
RCT → IPD meta-analysis → target-trial emulation → pharmacovigilance →
temporal initiation timing. Each intent yielded directly-claim-
relevant evidence; the judge produced 31 contradicts vs 11 supports,
correctly settling on `fail`.
