Forge, after the trim

A natural-language brief becomes a runnable agentic system. The model declares the shape; deterministic code does the load-bearing work; agents fill in the bodies; a sandbox checks the result. One graph, one straight line through it.

Reflects the forge-simplify branch — the self-correction loop, two advisory heads, and the dead code are gone (−2,313 lines). Companion to The Agentic Dialect.

Forge turns one sentence — "summarise the document into three bullets", "classify a support ticket and route it" — into a small, typed, dialect-conforming package you can run. It is not a chatbot and not an app: it builds a function, refusing anything with an external control loop at the door. The recurring idea underneath every stage is one line: the model proposes, deterministic code disposes.

model call (a head) deterministic code (no model) sandboxed execution

One graph, one straight line. There are no back-edges now: the redesign loop and the self-correction loop were removed. Every branch that remains is a stop-here decision (design-only, render-only, build-only), never a cycle.

1The one idea: the model proposes, code disposes

Generating a working program from a sentence is unreliable if you let a model do the whole thing. Forge's answer is to give the model only the jobs it is good at — naming the parts and writing a small typed body — and hand every load-bearing decision to deterministic code that can be tested and cannot hallucinate.

What the model does

Restates the brief, judges its shape, names the areas and the nodes, declares each node's inputs and outputs, and writes each node's body. It works in small, flat, structured answers — the kind a small local model fills reliably.

What deterministic code does

Matches the declared inputs to outputs into a graph, diagnoses every structural flaw, compiles and validates the spec, renders the whole package, and checks each authored body against its contract. No model writes the final wiring or the final files.

So a run is a relay: the model declares, code assembles and checks, the model corrects what code flagged, code renders, the model writes bodies, code gates them, and a sandbox runs the result. Wherever a check can be deterministic, it is — and where the model is asked for something, the answer is immediately verified by code rather than trusted.

2The pipeline, stage by stage

Eleven nodes in a single pydantic-graph. graph.py is the only engine-aware file; every stage below it is an engine-free worker. The three colours match the legend above — a model call, deterministic code, or sandboxed execution.

StageKindWhat it does
UnderstandheadRestates the brief as a problem: purpose and boundaries (in / out).
AssessheadThe fitness gate. Is the brief realisable as a function? A non-function shape (app / agent / service) is refused at the door with a concrete reshape — never quietly built wrong.
FrameheadSplits the problem into its 2–4 big concerns (areas).
Decomposeheads + detLists the jobs per area, then types each node — its kind and its named consumes / produces. CompiledetAssembles the validated SystemSpec. Canonicalises names, runs the deterministic per-area coverage check, keeps single-writer / dangling / orphan checks as fail-loud backstops.
ReviewdetA plain gate: proceed to render, or stop here for a design-only run. (The per-area coverage check already ran inside Decompose.)
RenderdetCode — not a model — writes the package: models, prompts, Deps, graph wiring, a smoke test, and a __main__ CLI. Node bodies are left as NotImplementedError holes.
VerifydetA cheap render-stage check: the package parses, imports, assembles, and matches the recipe.
Buildheads + detFor each hole, an agent drafts the body; static gates check it; on a failure the agent repairs it with the exact violated obligation. No model-written code runs here.

With dest=None it is design-only — understand through review, returning the validated spec without writing a file. stop_after ∈ {design, render, build, audit} caps it at each superset. Those are the only branches; each is a clean stop, not a loop.

3The nine heads

Every model call is one named head — a prompt plus a small typed output schema, called through a shared runner with an explicit model=. There are nine, in three groups. A simple five-node system makes on the order of twenty-odd calls; the design heads dominate the count.

GroupHeadsJob
Designunderstand, fitness, frame, list_jobs, type_nodeName the problem, judge its shape, and declare the node board — kinds and typed inputs/outputs.
Authoringbuild_draft, build_repairWrite each node body against its exact contract; repair it when a static gate objects.
Audit (advisory)requirements, criticDoes the built system serve the brief? What is missing or faked? These inform — they never decide whether the build passed.
Two heads were removed in the trim plan_manager (an LLM "does this plan serve the goal?" that looped the whole design back to Frame) and component_manager (an advisory "does this body do its job?" run after every gate-valid body). Both were prompt-hope on top of deterministic checks that already decided the outcome. The cheap deterministic coverage check they sat beside was kept.

4The build gates: a suggestion becomes a guarantee

The authoring stage is the fragile part — a model writing Python. Forge's discipline is that every instruction in the build prompt has a matching deterministic check that verifies the model obeyed it. A body is only accepted when it passes all of them; otherwise the exact violated obligation is fed back and the body is repaired, bounded by an attempt cap.

  • contract — reads, writes, and returns only what the node declared. No dynamic or bulk state access. Catches a hallucinated field name at build time, never as a runtime error.
  • coverage — actually reads every declared input and sets every declared output. Catches a body that quietly drops its input or never produces its output.
  • deps — touches only the dependencies the rendered Deps actually provides. A node that needs a resource the system never declared fails loud here, rather than inventing one.
  • purity — no process control, raw files, sockets, clock, or randomness; a network client only when the node declared it (and then only behind the container). Keeps generated code deterministic and safe to run.
  • fail-loud — no broad except that swallows the error. The house rule, enforced on generated code: a system that runs but does the wrong thing is worse than one that stops.

5The sandbox: the one place generated code runs

Model-written code never executes in the forge process. It runs only in the Audit stage, behind a sandbox Port. Podman is the default — host-isolated, read-only mount, scrubbed environment, memory and process caps; a pure node runs with no network, a declared-network node keeps the isolation but is allowed onto the net. A no-container subprocess fallback exists for pure nodes, but it is not host-isolated, so it refuses to run network code.

This matters for what audits cleanly A brief whose system fetches from the network (say, "summarise these URLs") can only be audited under Podman. Without the container image built, its tests cannot run, and the audit reports the system as incomplete even when every body was authored correctly — a backend limitation, not a fault in the generated code.

6What the trim changed

The simplification removed machinery that a personal tool did not need — roughly a third of the module — without changing what forge builds on the happy path.

RemovedWhy it went
The self-correction loopBest-build tracking, a regression guard, failure attribution, and re-materialising the best build onto disk — all to retry a red audit a couple of times. The largest, most intricate subsystem, for marginal gain. Audit is now a single honest pass.
The component_manager headAdvisory-only: it never decided whether a body was accepted (the static gates did). It roughly doubled the build-stage model calls for a nudge.
The review_plan loopAn LLM plan-approval that looped the whole design back to Frame. The deterministic per-area coverage check — the genuinely useful part — was kept.
Dead codeA human-gate render path that was never reached, a checkpoint-cap field never set, and a return-type knob always fixed to str.
Measured, not assumed The change kept every test green and did not regress the end-to-end success rate. On the one brief that exercised the removed loop directly, the build-and-audit path ran 2.3× faster for an identical outcome — the self-correction loop had been rebuilding and re-auditing without fixing anything.

7What "works well" honestly means today

Forge reliably produces a running system when the brief is simple enough that its nodes need little or no hand-written body. Where it is still unreliable is not the machinery that was trimmed — it is two things upstream and underneath, and they are the next work, not this one.

Design-loop stochasticity

Decompose needs the model to re-use produced-variable names exactly so code can match them. A small model that misses a few names burns the repair rounds and the design fails loud. It is non-deterministic run to run — the single biggest reliability lever, and the honest reason two runs of the same brief can differ.

The sandbox boundary

Network briefs need Podman to audit at all. On a machine without the container image, a whole class of otherwise-correct systems is reported incomplete simply because their tests cannot be run.

Run it

andamentum-forge build "<brief>" --model <id> --sandbox podman for the full design → render → author → audit pipeline, or design for a validated spec only. --stop-after {render,build,audit} caps the run; --verbose shows a live dashboard. There is no default model and no env-var fallback — the model is always explicit. The whole graph is testable with no live model and no container via stub Ports.
